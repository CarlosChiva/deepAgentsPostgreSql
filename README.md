# DeepAgent PostgreSQL

A Dockerized FastAPI chat API backed by PostgreSQL with [DeepAgents](https://deepagents.ai) / [LangGraph](https://langchain.github.io/langgraph/) for conversational AI.

## 1. Project Overview

**deepagent-postgres** is a production-ready conversational microservice. It exposes a REST API (`POST /chat`, `GET /chat/{thread_id}`) that routes messages to an agent built with DeepAgents and LangGraph. Agent state (conversation history + tool execution results) is persisted in PostgreSQL via `PostgresSaver`, enabling multi-threaded, resuming conversations.

- **API layer:** FastAPI (ASGI) with optional SSE streaming
- **Agent engine:** DeepAgents (`create_deep_agent()`) on top of LangChain 1.x + LangGraph
- **Persistence:** PostgreSQL 16 with pgvector (via `langgraph-checkpoint-postgres`)
- **Packaging:** Docker Compose for orchestration, `uv` for dependency management

## 3. Quick Start

```bash
# 1. Clone the project
git clone https://github.com/<you>/deepagent-postgres.git
cd deepagent-postgres

# 2. Copy example env file and set your LLM key
cp .env.example .env
# Edit .env and uncomment + set either OPENAI_API_KEY or ANTHROPIC_API_KEY

# 3. Start all services
docker compose up --build -d

# 4. Verify everything is healthy
curl -s http://localhost:8000/health | jq
```

The API is now live at **http://localhost:8000**. API docs are at **http://localhost:8000/docs**.

## 6. API Documentation

The API docs (Swagger UI) are available at **http://localhost:8000/docs**. The two main endpoints are documented below with curl examples.

### POST /chat — Send a Message

Sends a message to the agent and returns the reply. Set `stream=true` in the body for SSE streaming.

```bash
# Standard JSON response
curl -s http://localhost:8000/chat \
  -H "Content-Type: application/json" \
  -d '{
    "message": "Tell me about Docker multi-stage builds.",
    "thread_id": "abc-123"
  }' | jq
```

```bash
# SSE streaming — pipe to a viewer or terminal
curl -N http://localhost:8000/chat \
  -H "Content-Type: application/json" \
  -d '{
    "message": "Explain LangGraph checkpoints.",
    "thread_id": "abc-123",
    "stream": true
  }'
```

**Request body (`ChatRequest`)**

| Field | Type | Required | Default | Description |
|-------|------|----------|---------|-------------|
| `message` | `str` | ✅ | — | User input |
| `thread_id` | `str` | ❌ | Generated UUID | Conversation thread identifier |
| `stream` | `bool` | ❌ | `false` | Enable SSE streaming |

**Response (`ChatResponse`)**

```json
{
  "thread_id": "abc-123",
  "reply": "Docker multi-stage builds...",
  "timestamp": "2025-01-15T10:30:00"
}
```

### GET /chat/{thread_id} — Get Conversation History

Retrieves the full message history for a given thread from PostgreSQL.

```bash
curl -s http://localhost:8000/chat/abc-123 | jq
```

**Response (`ChatHistoryResponse`)**

```json
{
  "thread_id": "abc-123",
  "messages": [
    { "role": "user",    "content": "Tell me about Docker multi-stage builds.", "timestamp": "..." },
    { "role": "assistant","content": "Docker multi-stage builds allow...", "timestamp": "..." }
  ]
}
```

## 7. Health Endpoints

FastAPI applications ship with `/health` and `/ready` probes that Kubernetes or Docker Compose health checks can consume.

| Endpoint | Method | Purpose |
|----------|--------|---------|
| `/health` | GET | liveness probe — checks PostgreSQL connectivity and checkpointer status. Returns `200` when the DB is reachable and the checkpointer is initialized. |
| `/ready` | GET | readiness probe — returns `200` once the server is fully started. Returns `503` during startup or after a failed initialization. |
| `/docs` | GET | Swagger UI (auto-generated from FastAPI) |
| `/openapi.json` | GET | Raw OpenAPI schema |

Example:

```bash
# Liveness
curl -s http://localhost:8000/health | jq

# Readiness
curl -s http://localhost:8000/ready | jq
```

## 8. Configuration — Choosing an LLM Provider

The agent will pick a model provider based on which API keys are set:

```
OPENAI_API_KEY  set  → OpenAI (e.g., "gpt-4o", "gpt-4o-mini")
ANTHROPIC_API_KEY set → Anthropic Claude (e.g., "claude-sonnet-4-5-20250929")
Neither set     → Error on startup
```

1. **OpenAI:** Uncomment and set `OPENAI_API_KEY` in `.env`:
   ```env
   OPENAI_API_KEY=sk-proj-your-key-here
   LLM_MODEL=gpt-4o
   ```

2. **Anthropic:** Uncomment and set `ANTHROPIC_API_KEY` in `.env`:
    ```env
    ANTHROPIC_API_KEY=sk-ant-your-key-here
    LLM_MODEL=claude-sonnet-4-5-20250929
    ```

3. **ChatOllama (local):** Point to a local or remote ChatOllama/Ollama instance:
    ```env
    CHATOLLAMA_BASE_URL=http://localhost:11434
    LLM_MODEL=llama3.2
    ```
    The ChatOllama/OLLAMA service must be running and the model (`ollama pull llama3.2`) must be available locally.

> **Note:** In production, never commit your `.env` or `.env.production` to version control. Add them to `.gitignore`.

## 10. Troubleshooting

| Problem | Likely Cause | Fix |
|---------|-------------|-----|
| **DB connection refused** | PostgreSQL container not running yet | `docker compose up postgres`; wait for `pg_isready` |
| **FATAL: database "chatdb" does not exist** | DB name in `.env` does not match the Docker init | Ensure `POSTGRES_DB` matches |
| **FATAL: password authentication failed** | Wrong `POSTGRES_PASSWORD` | Confirm `POSTGRES_PASSWORD` in `.env` matches the compose config |
| **API key not set / model not selected** | Neither `OPENAI_API_KEY` nor `ANTHROPIC_API_KEY` in `.env` | Set at least one key in `.env` |
| **400 Bad Request on POST /chat** | Missing `message` field in request body | Verify the request body has the required `message` string field |
| **Port 8000 already in use** | Another process is bound to `APP_PORT` | Change `APP_PORT` in `.env`, then `docker compose down && docker compose up --build` |
| **Port 5432 conflict** | Host-level PostgreSQL or another service uses the default port | Change `POSTGRES_PORT` in `.env` and restart the compose stack |
| **Tables already exist on second `docker compose down -v && docker compose up`** | `PostgresSaver.setup()` tries to recreate tables | The app should catch `table already exists` on `.setup()` — verify the try/except is present in `app/database.py` |
| **Empty response history on GET /chat/{thread_id}** | Thread never received any `POST /chat` call, or DB volume was wiped | Verify the thread was previously created; ensure the named volume is not removed (`docker compose down -v`) |
